Graph Name System

Names that resolve through the
Nostr social graph.

DNS maps names to servers. NIP-05 maps names to public keys. GNS resolves identities through graph relationships — so the same name can mean different people depending on where you stand in the graph.

DNSname → server
NIP-05name → pubkey
GNS(name, graph position) → pubkey

The scarce resource isn't the name — it's your position in the graph.

The same person is reachable through many paths, and each path is provenance: it tells you how you discovered them.

barbara@alex.michael.nostr
  • barbara — the target name
  • alex.michael — the graph path (you → michael → alex)
  • .nostr — optional namespace TLD

Read right-to-left from your namespace: find michael among your follows, alex among michael's follows, barbara among alex's. A bare barbara is a compressed direct lookup.

    How a name becomes a label

    A profile is eligible for GNS only if its kind-0 name normalizes to a non-empty label.

    Normalization

    Lowercase, then keep only ASCII a–z and 0–9.

    "Michael Saylor"→michaelsaylor
    "MICHAEL-SAYLOR"→michaelsaylor
    "⚡Michael⚡"→michael
    "FiatJaf"→fiatjaf

    Membership & ambiguity

    A label x belongs to namespace y iff y's follow list contains a pubkey that normalizes to x.

    If multiple followed pubkeys collide on a label, it is ambiguous and must not resolve — the resolver returns every alternate, fully-labelled path so a client can disambiguate.

    Names are interpreted relative to a location in the graph, so they need not be globally unique:

    john.company john.friend john.bitcoin

    All can coexist.

    What the resolver gives you

    ⛓

    Full provenance

    Every hop carries its follow event id and the relays it was seen on — discovery you can audit.

    🛰

    Relay attribution

    A minimal per-relay websocket client so each follow event is attributed to the relays that served it.

    ∅

    No new NIP

    Built entirely on existing kind-0 profiles and kind-3 contact lists. Relays stay the source of truth.

    ♻

    Freshness cache

    A TTL "freshness window" over follow lists and profiles — the public-good caching dynamic.

    ✓

    Verified events

    Schnorr signatures checked on every fetched event; NIP-01 ids recomputed. Clients can re-verify.

    ◍

    Live dashboard

    A dependency-free dashboard draws the resolved path as avatar bubbles with per-hop relays and ids.

    A Web of Names, not a Web of Trust

    GNS turns the Nostr social graph into a decentralized naming layer. Trust may emerge from the graph, but the primary purpose is discovery and naming. If an indexer disappears, another can rebuild entirely from relay data. A client resolves barbara@alex.michael.nostr to a pubkey, encrypts to it, and delivers — the user never sees a public key.
